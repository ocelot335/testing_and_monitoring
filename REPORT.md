## 1. Доработка кода сервиса
В код сервиса(ml_service/app.py) были добавлены следующие обработки исключений и проверки:
1. Невалидный run_id: В ручке /updateModel вызов MODEL.set(run_id) обернут в try-except. Если модель не найдена в MLflow, сервис не падает с 500 ошибкой, а корректно возвращает 400 Bad Request с описанием ошибки.
2. Проверка наличия всех фичей: В ручке /predict после формирования датафрейма добавлена проверка df.isnull().values.any(). Если клиент не прислал признаки, которые требуются текущей версии модели, возвращается ошибка 422 Unprocessable Entity с указанием недостающих колонок.
3. Ошибка на инференсе: Вызов model.predict_proba(df) обернут в try-except. Если входные данные имеют неверный формат(например, текст вместо числа), из-за чего падает пайплайн scikit-learn, сервис перехватывает исключение и возвращает 400 Bad Request.

## 2. Написание тестов на сервис
Для тестирования использовался фреймворк pytest и TestClient из FastAPI. Для изоляции тестов от сети и внешнего MLflow-сервера применялся unittest.mock.patch, который подменял загрузку реальной модели на легковесную мок-заглушку DummyModel.

Были написаны тесты(tests/test_app.py) покрывающие все 4 критерия из задания:

1. Функции предобработки данных: тест test_to_dataframe проверяет, что Pydantic-схема запроса корректно конвертируется в pandas.DataFrame, а также проверяет логику фильтрации признаков, что оставляются только те колонки, которые требует текущая модель.
2. Успешность загрузки и инференса модели: тест test_update_model_success имитирует скачивание новой модели по run_id и проверяет успешный возврат 200 OK. В DummyModel реализован метод predict_proba, что позволяет тестировать успешный расчёт вероятностей предсказания.
3. Логика хэндлеров и валидность запросов: тест test_predict_missing_features проверяет валидацию в /predict: если переданы не все обязательные для модели признаки, возвращается HTTP-код 422 Unprocessable Entity. Тест test_update_model_invalid_run_id проверяет обработку невалидного идентификатора: при ошибке MLflow сервис возвращает 400 Bad Request, а не падает с 500 ошибкой.
4. Работа всего сервиса целиком (сервис поднимается и отдаёт предсказание): тест test_predict_success выполняет сквозную проверку работы приложения. Использование TestClient(app) в контекстном менеджере запускает полный жизненный цикл FastAPI(то есть lifespan). Тест проверяет, что HTTP POST-запрос на /predict успешно проходит слои валидации Pydantic, доходит до бизнес-логики, обращается к интерфейсу загруженной модели(в данном случае изолированной DummyModel), формирует правильный ответ и возвращает 200 OK.

## 3. Добавление логирования метрик
Для сбора метрик был выбран Prometheus и prometheus_client. В FastAPI-приложение добавлен специальный хэндлер /metrics (через app.mount("/metrics", metrics_app)).

В коде реализовано логирование следующих блоков метрик:

1. Технические метрики работы сервиса:
   - Число запросов и статистика по кодам ошибок: http_requests_total(тип Counter с лейблами endpoint и http_status, включая обработку 400, 422 и 503 ошибок).
   - Время ответа сервиса: http_request_duration_seconds(тип Histogram).
   - Потребление ресурсов(CPU, RAM) собирается автоматически базовыми метриками prometheus_client при инициализации.
2. Метрики, связанные с входными данными:
   - Время предобработки данных: data_preprocess_duration_seconds (тип Histogram).
   - Статистика по значениям фичей: для численных признаков используется feature_values(тип Histogram), для категориальных(строковых) - feature_categorical_total(тип Counter с сохранением значения в лейбл category_value).
3. Метрики работы модели:
   - Время инференса(вызова predict_proba): model_inference_duration_seconds(тип Histogram).
   - Статистика по предсказаниям: распределение вероятностей пишется в model_probabilities(тип Histogram), а итоговые классы - в model_predictions_total(тип Counter).
4. События и информация о модели:
   - Обновление модели считается счётчиком model_updates_total(с лейблом status: success/failed).
   - Информация о текущей загруженной модели(тип используемого алгоритма model_type, идентификатор run_id и требуемые фичи features) хранится в метрике current_model(тип Info). Данные обновляются при старте сервиса и при каждом успешном вызове /updateModel.

### Расчёт перцентилей(75%, 90%, 95%, 99%, 99.9%):
Расчёт перцентилей осуществляется на стороне Grafana с помощью функции histogram_quantile. Для обеспечения высокой точности интерполяции при расчёте этих метрик, в коде сервиса явно задан массив TIME_BUCKETS(от 5 мс до 10 сек), так как дефольные бакеты используются для быстрых сервисов, а не для ML-инференса

## 4. Создание дашборда в Grafana
В Grafana создан единый дашборд с логической компоновкой панелей(сверху вниз: от бизнес-показателей к техническим деталям). Добавлена дашборд-переменная $instance для динамической фильтрации метрик по конкретному IP-адресу сервера. 

Реализовано 13 визуализаций, которые полностью покрывают все 4 требуемых блока:

1. Технические метрики работы сервиса:
   - HTTP Запросы(RPS): Time series график скорости поступления запросов с разбивкой по HTTP-статусам.
   - Время ответа сервиса: Time series график, отображающий 5 перцентилей(75, 90, 95, 99, 99.9) через функцию histogram_quantile.
   - Потребление ресурсов: два графика Time series для мониторинга использования оперативной памяти(в байтах) и утилизации CPU(в процентах).
2. Метрики, связанные с данными:
   - Время предобработки данных: Time series график с 5 перцентилями времени выполнения функции конвертации признаков.
   - Среднее числовых фичей: динамическая панель, вычисляющая среднее арифметическое всех поступающих численных признаков за выбранный период времени.
   - Статистика категориальных фичей: горизонтальный Bar gauge(с сортировкой по убыванию), показывающий частоту встречаемости различных строковых значений во входящих данных.
3. Метрики, связанные с работой модели:
   - Уверенность модели: Heatmap(тепловая карта) распределения вероятностей предсказаний модели.
   - Динамика предсказаний: Time series(в режиме Stacked Bars), отображающий количество предсказаний класса 0 и 1 во времени.
   - Время инференса модели: Time series график с 5 перцентилями времени выполнения метода predict_proba.
4. Метрики информации по модели и её обновлениям:
   - Текущая ML-модель: Table с информацией о загруженном run_id, типе алгоритма и требуемых признаках.
   - Всего обновлений: Stat панель, показывающая общее количество успешных(зеленый) и провальных(красный) попыток обновления модели.
   - Обновления модели: ступенчатый Time series график, фиксирующий моменты вызова эндпоинта /updateModel.

## 6. Мониторинг дрифта данных и предсказаний с помощью Evidently
Сбор данных и генерация метрик реализованы в асинхронном фоновом процессе, чтобы не блокировать обработку пользовательских запросов:
1. Накопление данных: при каждом успешном вызове /predict входные признаки и ответ модели сохраняются в потокобезопасные списки в памяти сервиса.
2. Формирование референса: первая накопленная партия запросов( >10 штук) автоматически фиксируется как референс. Все последующие партии используются как current_data для сравнения с reference_data.
3. Фоновая отправка: при старте приложения запускается корутина, которая раз в минуту просыпается, формирует отчет и отправляет его на сервер(http://158.160.2.37:8000/projects/019d3afe-3198-7a78-b501-032ef27b9191/reports).
